# Módulo Padrões de Projeto GoF(s) Estruturais

## 1. Introdução

Os padrões de projeto Estruturais tratam da composição de classes e objetos para formar estruturas maiores. Esses padrões ajudam a garantir que, ao construir sistemas complexos, os componentes individuais possam ser organizados de forma a colaborar de maneira eficiente e flexível. Eles focam em como classes e objetos podem ser combinados para formar novas funcionalidades sem grandes mudanças no código existente.

Os padrões estruturais são essenciais quando se busca criar sistemas robustos e flexíveis, que possam ser facilmente adaptados e mantidos. Eles fornecem soluções elegantes para a organização de objetos e classes, permitindo a construção de estruturas mais complexas e modulares. Dessa forma, o uso de padrões estruturais facilita a reutilização de código e a manutenção do sistema ao longo do tempo, promovendo uma arquitetura de software mais sólida e bem definida.

## 3. Padrão de Projeto Estrutural - Adapter

### 3.1. Base 

O padrão Adapter é utilizado para permitir que classes com interfaces incompatíveis possam trabalhar juntas. Ele atua como um conversor, adaptando a interface de uma classe para outra interface esperada pelos clientes. Esse padrão é especialmente útil quando se deseja utilizar uma classe existente mas cuja interface não coincide com a esperada pelo sistema.

### 3.3. Implementação do Adapter

```java
public interface ConversaInterface {
    public void criar();
    public void ler(String msg);
}

public class ConversaService {
    public void method() {
        System.out.println("Método do ConversaService chamado");
    }
}

public class ConversaAdapter implements ConversaInterface {
    private final ConversaService conversaService;

    public ConversaAdapter(ConversaService conversaService) {
        this.conversaService = conversaService;
    }

    @Override
    public void criar() { conversaService.method() }

    @Override
    public void ler(String msg) {
        System.out.println("Lendo mensagem " + msg);
        conversaService.method();
    }
}

public class Main {
    public static void main(String[] args) {
        ConversaService ConversaService = new ConversaService();
        ConversaInterface client = new conversaAdapter(conversaService);

        client.criar();
        client.ler("Teste");
    }
}
```

## 4. Padrão de Projeto Estrutural - Facade

### 4.1. Base 

O padrão Facade oferece uma interface simplificada para um conjunto de interfaces em um subsistema. Ele define uma interface de alto nível que torna o subsistema mais fácil de usar. O objetivo é esconder a complexidade do subsistema e fornecer uma interface mais acessível para o cliente.

## 5. Padrão de Projeto Estrutural - Composite

### 5.1. Base

O padrão de projeto Composite é um dos padrões estruturais do GoF (Gang of Four) que permite compor objetos em estruturas de árvore para representar hierarquias parte-todo. Este padrão trata de um problema recorrente no design de sistemas onde objetos individuais e suas composições precisam ser tratados de maneira uniforme.

### 5.3. Implementação do Composite

```java
// Interface Componente
interface Usuario {
    void cadastrarUsuario();
}

// Classe Folha
class UsuarioNaoCadastrado implements Usuario {
    public void cadastrarUsuario() {
        System.out.println("Usuário não cadastrado realizando cadastro...");
    }
}

// Interface Adicional para Usuários Cadastrados
interface UsuarioCadastrado extends Usuario {
    void deletarUsuario();
    void editarUsuario();
    void visualizarUsuario();
}

// Classe Composto
class UsuarioCadastradoImpl implements UsuarioCadastrado {
    public void cadastrarUsuario() {
        System.out.println("Usuário já cadastrado realizando novo cadastro...");
    }

    public void deletarUsuario() {
        System.out.println("Deletando usuário cadastrado...");
    }

    public void editarUsuario() {
        System.out.println("Editando usuário cadastrado...");
    }

    public void visualizarUsuario() {
        System.out.println("Visualizando usuário cadastrado...");
    }
}

// Código Cliente
public class Cliente {
    public static void main(String[] args) {
        // Criando instâncias de usuários
        Usuario usuarioNaoCadastrado = new UsuarioNaoCadastrado();
        UsuarioCadastrado usuarioCadastrado = new UsuarioCadastradoImpl();

        // Operações de um usuário não cadastrado
        usuarioNaoCadastrado.cadastrarUsuario();

        // Operações de um usuário cadastrado
        usuarioCadastrado.cadastrarUsuario();
        usuarioCadastrado.deletarUsuario();
        usuarioCadastrado.editarUsuario();
        usuarioCadastrado.visualizarUsuario();
    }
}
```

## 6. Conclusão

A criação do artefato sobre padrões de projeto GoF estruturais foi essencial para o desenvolvimento do projeto de aprendizado da equipe. Através da criação dos diagramas, foi possível explorar com profundidade os padrões adotados e compreender seus benefícios e limitações.

No que tange o uso do padrão Facade, ele simplificou a complexidade do sistema ao introduzir uma interface de alto nível (FacadePrincipal e FacadeUsuario), de forma que foi possível encapsular a interação com diversos subsistemas, facilitando o uso e a manutenção do código. Esta implementação também destacou desafios como garantir que todas as funcionalidades essenciais fossem expostas de forma clara e eficiente. Para isto, foi necessário realizar uma análise cuidadosa das dependências e interações entre os componentes. Além disso, este padrão possibilitou uma arquitetura mais modular e flexível, que pode ser facilmente adaptada para futuras expansões e modificações.

Já relativo ao padrão Adapter, ele se mostrou crucial para a integração de classes com interfaces incompatíveis. A criação do ChatAdapter permitiu que o ChatService fosse utilizado de maneira coerente com a interface ChatInterface, demonstrando a capacidade do Adapter de promover a reutilização de código existente sem a necessidade de grandes refatorações. Este padrão facilitou a adaptação de componentes e promoveu a interoperabilidade entre módulos previamente isolados. Um de seus maiores desafios esta em garantir que a adaptação mantenha a integridade e o desempenho do sistema, o que exigiu uma análise rigorosa.

Por fim, a implementação do padrão Composite revelou-se fundamental para lidar com a hierarquia parte-todo no gerenciamento de usuários. A capacidade de tratar objetos individuais e composições de maneira uniforme simplificou o código e melhorou a escalabilidade do sistema. A estrutura hierárquica permitiu uma maior flexibilidade na manipulação de diferentes tipos de usuários (UsuarioNaoCadastrado e UsuarioCadastradoImpl), promovendo um design mais intuitivo e fácil de manter. Sua complexidade foi encontrada na implementação inicial que exigiu um entendimento profundo das relações entre os componentes e a criação de uma interface comum que pudesse ser compartilhada de maneira eficaz.
